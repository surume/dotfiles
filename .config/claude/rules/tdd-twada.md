# t_wada 流 TDD

テストコードを書くときは以下の原則に従う。

## サイクル

1. **Red**: まず失敗するテストを書く。テストが失敗することを確認する。
2. **Green**: テストを通す最小限のコードを書く。過剰な実装はしない。
3. **Refactor**: テストが通った状態でリファクタリング。テストは常にグリーンを維持。

## テストの書き方

- テスト名は日本語で「〜すること」「〜の場合、〜を返すこと」と書く
- Arrange-Act-Assert (AAA) パターンで構造化する
- 1テスト1アサーション を原則とする
- テストの独立性を保つ（テスト間で状態を共有しない）
- テストファーストで書く（実装より先にテストを書く）

## テストの分類

- 小さいテスト（単体）: 外部依存なし、高速
- 中くらいのテスト（結合）: DB やファイルシステム等を含む
- 大きいテスト（E2E）: システム全体を通す

## アンチパターンを避ける

- テストにロジックを書かない（if/for/try-catch をテスト内に書かない）
- テストヘルパーを過度に抽象化しない
- モックを使いすぎない（本物を使えるなら本物を使う）
- テストの実行順序に依存しない
